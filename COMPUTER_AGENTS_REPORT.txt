1. HERMES AGENT (Nous Research)
================================================================================
Creator:    Nous Research
Year:       2025
Website:    hermes-agent.nousresearch.com
GitHub:     github.com/NousResearch/hermes-agent (215K+ stars)

What it does:
  The self-improving AI agent built by Nous Research. It creates skills from
  experience, improves them during use, nudges itself to persist knowledge,
  searches its own past conversations, and builds a deepening model of who you
  are across sessions. Runs anywhere -- VPS, GPU cluster, or serverless.

Architecture:
  - Local-first gateway architecture with 6 terminal backends (local, Docker,
    SSH, Singularity, Modal, Daytona)
  - Agent loop with tool-calling LLMs
  - Pluggable memory system with 8 external providers (Honcho, Hindsight, Mem0,
    OpenViking, Supermemory, etc.)
  - Skills system for procedural memory (create, self-improve, reuse)
  - Multi-channel messaging gateway (Telegram, Discord, Slack, WhatsApp, Signal)
  - FTS5 session search with LLM summarization for cross-session recall
  - MCP integration for extending tool capabilities

Key Benefits:
  + Closed learning loop: autonomous skill creation and self-improvement
  + Runs anywhere -- not tied to laptop; works on $5 VPS or GPU clusters
  + 8 pluggable memory providers with cross-session recall
  + Multi-channel messaging gateway (29+ platforms)
  + Built-in cron scheduler for unattended automations
  + Research-ready: batch trajectory generation, RL training support
  + MIT licensed, fully open source

Key Disadvantages:
  - Complex setup with many configuration options
  - Large codebase (82% Python, 15% TypeScript)
  - Requires API keys for LLM providers (not self-hosted models by default)
  - Memory providers can be confusing to choose from
  - Still relatively new (July 2025), ecosystem still maturing

Open Source: Yes -- MIT License
Platforms:   macOS, Linux, Windows (WSL2), Android

2. OPEN INTERPRETER
================================================================================
Creator:    OpenInterpreter (originally Killian Lucas, now OpenAI-affiliated)
Year:       2023 (Python), 2025 (Rust rewrite)
GitHub:     github.com/openinterpreter/openinterpreter (65K+ stars)

What it does:
  A coding agent optimized for low-cost models. Fork of OpenAI's Codex with
  focus on emulating the agent harness that gets best performance from cheap
  models. Lets LLMs run code locally via natural language.

Architecture:
  - Central OpenInterpreter class orchestrating message-passing architecture
  - Computer class for code execution through run() method
  - LLM class using litellm for model provider abstraction
  - Streaming chat interface with chunk-based response format
  - Terminal backends for different execution environments
  - QA skill for browser automation via agent-browser
  - Native sandboxing on macOS, Linux, Windows
  - Agent Client Protocol (ACP) support for editor integration

Key Benefits:
  + Natural language to code execution pipeline
  + Supports 30+ LLM providers via litellm
  + Built-in QA skill for web and native app testing
  + Rust rewrite (based on Codex) for performance
  + Works with low-cost models effectively
  + ACP support for IDE integration
  + Local execution -- no cloud dependency

Key Disadvantages:
  - Security risks with unrestricted local code execution
  - Requires careful sandboxing for safety
  - Rust rewrite is newer and may have fewer features than Python version
  - Primarily a coding tool, not a general computer-use agent
  - No built-in memory/persistence across sessions

Open Source: Yes -- Apache 2.0 License
Platforms:   macOS, Linux, Windows

3. OS-COPILOT (Shanghai AI Lab)
================================================================================
Creator:    Shanghai AI Lab, East China Normal University, Princeton, HKU
Year:       2024
Paper:      arxiv.org/abs/2402.07456 (ICLR 2024 Workshop)
GitHub:     github.com/OS-Copilot/OS-Copilot (1.8K stars)

What it does:
  A framework for building generalist agents capable of interfacing with
  comprehensive OS elements: web, code terminals, files, multimedia, and
  third-party applications. Created FRIDAY, a self-improving embodied agent.

Architecture:
  - Modular framework with universal interface for OS interactions
  - FRIDAY agent built on top with self-directed learning
  - Configurator with self-learning module
  - Curriculum-based self-directed learning (generates tasks from easy to hard)
  - Trial-and-error skill accumulation
  - Vision mode added (2024) for screenshot-based interaction
  - Frontend interface for user interaction
  - Linux and macOS support

Key Benefits:
  + Self-directed learning -- accumulates skills without human supervision
  + Outperformed baselines by 35% on GAIA benchmark
  + Generalizes to unseen applications via accumulated skills
  + Learns to control Excel and PowerPoint with minimal supervision
  + Modular framework for building custom agents
  + Research-backed with strong benchmark results

Key Disadvantages:
  - Limited to Linux and macOS (no Windows support)
  - Single-round conversation only (no multi-turn)
  - Relatively small community (20 contributors)
  - Not actively maintained (last push Sept 2024)
  - Self-learning can be slow and unpredictable

Open Source: Yes -- MIT License
Platforms:   Linux, macOS

5. ANTHROPIC COMPUTER USE (Claude)
================================================================================
Creator:    Anthropic
Year:       2024 (beta), improved through 2025-2026
Docs:       platform.claude.com/docs/agents-and-tools/tool-use/computer-use-tool

What it does:
  Claude can interact with desktop environments through screenshot-based
  computer use -- seeing screens, moving cursors, clicking buttons, and typing
  text. The first frontier AI model to offer computer use in public beta.

Architecture:
  - Screenshot-based perception (captures display, sends to vision model)
  - Mouse/keyboard control via API commands
  - Agent loop: Claude requests actions -> app executes -> returns screenshots
  - Containerized environment with virtual X11 display (Xvfb)
  - Lightweight desktop (Mutter + Tint2) for consistent GUI
  - Supports actions: click, type, scroll, drag, zoom, key combos
  - Computer_20251124: zoom into specific screen regions at full resolution
  - Text editor and bash tools for code operations

Key Benefits:
  + State-of-the-art on OSWorld screenshot-only category (14.9% -> 22%)
  + Works with any application -- no API/DOM dependencies
  + Self-correcting: retries when encountering obstacles
  + Rapidly generalizes from minimal training software
  + Zoom capability for precise small-element interaction
  + Available on API, Bedrock, and Vertex AI

Key Disadvantages:
  - Slow and often error-prone (still beta)
  - "Flipbook" view misses short-lived actions/notifications
  - Struggles with scrolling, dragging, zooming
  - Requires containerized/sandboxed environment
  - Cloud-only -- requires API calls (latency + cost)
  - No built-in memory or learning across sessions
  - Still far from human-level (14.9% vs 72.4% human on OSWorld)

Open Source: No -- API only (reference implementations are open)
Platforms:   Any (via API -- Linux sandbox recommended)

6. AUTO-GPT
================================================================================
Creator:    Significant Gravitas
Year:       2023
GitHub:     github.com/Significant-Gravitas/AutoGPT (185K+ stars)

What it does:
  Autonomous GPT-4 agent that can pursue goals with minimal human input.
  Originally a fully autonomous loop, now evolved into a visual workflow
  platform for building, deploying, and managing AI agents.

Architecture:
  - Originally: ReAct loop with thought -> action -> observation cycles
  - Memory: Embeddings via OpenAI text-embedding-ada-002 -> vector DB (Pinecone)
  - Summarization for context management
  - Now: Visual workflow builder (FastAPI + React + PostgreSQL)
  - Block-based system where each block performs a single action
  - ExecutionManager resolves dependencies, runs concurrent block executions
  - WebSocket for real-time monitoring
  - Plugin system (originally hook-based, now more structured)
  - Agent Protocol standard for interoperability

Key Benefits:
  + Pioneered the autonomous agent concept
  + Large community and ecosystem (185K+ stars)
  + Visual workflow builder for non-developers
  + Marketplace for pre-built agents
  + Agent Protocol for cross-platform compatibility
  + Active development with continuous updates

Key Disadvantages:
  - Original autonomous mode was unreliable (hallucinations, loops)
  - Pivot to workflow platform split the community
  - Complex architecture with many moving parts
  - Resource-intensive (API costs for GPT-4)
  - Platform version licensed under Polyform Shield (not fully open)

Open Source: Partially -- MIT for classic agent, Polyform Shield for platform
Platforms:   Web-based (platform), CLI (classic agent)

12. AIOS
================================================================================
Creator:    AIOS Research (agi-research)
Year:       2024
GitHub:     github.com/agiresearch/AIOS
Paper:      arxiv.org/abs/2403.16971 (COLM 2025)

What it does:
  An LLM Agent Operating System that embeds large language models into the OS,
  providing kernel-level services (scheduling, context management, memory
  management, storage, access control) for LLM-based agents.

Architecture:
  - Two-layer design: Application layer + Kernel layer
  - AIOS Kernel: scheduler, context manager, memory manager, storage manager,
    tool manager, access manager
  - System calls (AIOS syscalls) for agent resource requests
  - FIFO and Round Robin scheduling strategies
  - LRU-K eviction policy for memory management
  - Context switching with snapshot and restoration
  - AIOS SDK (Cerebrum) for agent development
  - Multiple deployment modes: Local, Remote, Personal, Virtual
  - Supports ReAct, Reflexion, AutoGen, OpenAGI, MetaGPT frameworks

Key Benefits:
  + OS-level resource management for agents
  + 2.1x faster execution via proper scheduling
  + Context switching enables concurrent agent execution
  + Memory management prevents resource exhaustion
  + Framework-agnostic -- works with multiple agent frameworks
  + Academic rigor (accepted at COLM 2025)
  + Multiple deployment modes for different use cases

Key Disadvantages:
  - Research-oriented, not production-ready
  - Complex architecture with many components
  - Overhead from kernel abstraction layer
  - Limited documentation and examples
  - Requires understanding of OS concepts
  - Small community

Open Source: Yes (custom license -- NOASSERTION)
Platforms:   Linux, macOS (Python-based)

CROSS-CUTTING ANALYSIS
================================================================================

COMMON PATTERNS ACROSS ALL AGENTS:
------------------------------------
1. LLM-as-Brain: Every agent uses an LLM for reasoning/planning
2. Tool Use: All agents extend LLM capabilities with external tools
3. Perception-Action Loop: Screenshot/parse -> reason -> act -> observe
4. Memory Systems: Most implement some form of persistent memory
5. Sandboxing: Nearly all use isolation for safe code execution
6. Multi-Model Support: Trend toward model-agnostic architectures
7. MCP Protocol: Emerging standard for tool integration (Hermes, UFO, OpenClaw)
8. Self-Improvement: Several agents learn from experience (Hermes, OS-Copilot)
9. Multi-Agent: Trend toward agent orchestration (UFO³, Devin, AIOS)

WHAT WORKS WELL:
-----------------
1. Hybrid GUI+API approaches (UFO²: 51% fewer LLM calls)
2. Screenshot-based perception (generalizable across applications)
3. Accessibility tree integration (structured UI understanding)
4. Demonstration-conditioned learning (OpenAdapt: 100% accuracy)
5. Sandboxed execution (E2B: 200ms startup, Firecracker isolation)
6. Custom ACIs for specific domains (SWE-agent: 64% improvement)
7. Speculative multi-action prediction (batch actions, reduce LLM calls)
8. Cross-session memory with knowledge graphs (Hindsight, Hermes)
9. RAG-enhanced learning from execution history (UFO² knowledge substrate)
10. Multi-channel delivery (Hermes, OpenClaw: reach users where they are)

WHAT DOESN'T WORK:
-------------------
1. Pure screenshot-only agents (14.9% on OSWorld vs 72.4% human)
2. Fully autonomous loops without guardrails (Auto-GPT original)
3. Single-round conversation (OS-Copilot FRIDAY limitation)
4. Platform-specific agents (UFO² Windows-only, no cross-platform)
5. Black-box models without explainability (most cloud agents)
6. Ignoring DOM/accessibility when available (pure vision is weaker)
7. Large context windows without compression (expensive, slow)
8. One-size-fits-all agents (specialization beats generalization currently)
9. No human-in-the-loop (unpredictable autonomous behavior)
10. Ignoring privacy/PII concerns (most agents lack scrubbing)

KEY ARCHITECTURAL DECISIONS:
------------------------------
1. LOCAL vs CLOUD: Local-first (Hermes, OpenClaw) vs cloud (Anthropic, E2B)
   -> Local-first wins for privacy, latency, cost; cloud wins for power
2. SCREENSHOT vs API vs DOM: Screenshot (CogAgent, OmniParser) vs DOM (UFO²)
   -> Hybrid approaches win (screenshot for unknown, API when available)
3. SINGLE vs MULTI-AGENT: Single (most) vs multi-agent (UFO³, Devin fleet)
   -> Multi-agent for complex tasks; single for simple ones
4. LEARNING APPROACH: Prompt engineering vs fine-tuning vs demonstration
   -> Demonstration-conditioned (OpenAdapt) shows strongest results
5. EXECUTION ENVIRONMENT: Direct (risky) vs sandboxed (safe) vs VM (isolated)
   -> Sandboxed with escape hatches is the pragmatic choice
6. MEMORY ARCHITECTURE: Stateless vs file-based vs vector DB vs knowledge graph
   -> Knowledge graph + vector DB hybrid provides best recall
7. ACTION PREDICTION: Single-action vs speculative multi-action
   -> Speculative multi-action (UFO²) reduces LLM calls by 51%

FEATURES TO STEAL FOR JARVIS
================================================================================

1. SKILL CREATION FROM EXPERIENCE (Hermes Agent)
   WHY: Agents that create and improve skills autonomously become more
   capable over time without retraining. Hermes' closed learning loop with
   skill self-improvement is the gold standard for agent evolution.

2. SPECULATIVE MULTI-ACTION PREDICTION (UFO²)
   WHY: Predicting multiple actions in a single LLM call reduces API costs
   by 51% and latency dramatically. JARVIS should batch action predictions
   rather than making one LLM call per action.

3. DEMONSTRATION-CONDITIONED INFERENCE (OpenAdapt)
   WHY: OpenAdapt proved that conditioning on human demos improves accuracy
   from 46.7% to 100% on macOS tasks. JARVIS should record user
   demonstrations and use them as runtime context for better action selection.

4. PICTURE-IN-PICTURE DESKTOP (UFO²)
   WHY: Running automation in a sandboxed virtual desktop prevents
   interference with user workflow. JARVIS should use PiP for background
   automation tasks while the user continues working.

5. PLUGGABLE MEMORY PROVIDERS (Hermes Agent)
   WHY: 8 different memory backends (Honcho, Hindsight, Mem0, etc.) let
   users choose the right memory for their use case. JARVIS should support
   swappable memory backends with a unified interface.

6. KNOWLEDGE SUBSTRATE / RAG FROM EXECUTION HISTORY (UFO²)
   WHY: Learning from past execution traces without model retraining is
   powerful. JARVIS should build a RAG index of past actions, docs, and
   demos to improve over time.

7. AGENT-COMPUTER INTERFACE DESIGN (SWE-agent)
   WHY: Custom interfaces designed for LLMs (not humans) improved
   performance 64% over generic shell access. JARVIS should design
   purpose-built interfaces for each tool/domain.

8. MULTI-CHANNEL GATEWAY (Hermes Agent / OpenClaw)
   WHY: Reaching users on Telegram, Discord, Slack, WhatsApp, etc. from a
   single gateway is essential for a personal assistant. JARVIS should
   support all major messaging platforms.

9. PII/PHI SCRUBBING (OpenAdapt)
   WHY: Privacy is non-negotiable for a personal assistant. OpenAdapt's
   Presidio-based scrubbing with before/after review is the right pattern.
   JARVIS should scrub sensitive data before any external API calls.

10. SANDBOXED CODE EXECUTION (E2B)
    WHY: E2B's Firecracker microVMs start in 200ms and provide complete
    isolation. JARVIS should use lightweight sandboxed environments for
    any code execution rather than running on the host.

11. CROSS-SESSION MEMORY WITH FTS5 SEARCH (Hermes Agent)
    WHY: Full-text search with LLM summarization enables recalling specific
    past conversations. JARVIS should index all sessions for search and
    provide LLM-summarized recaps of past interactions.

12. CURRICULUM-BASED SELF-DIRECTED LEARNING (OS-Copilot/FRIDAY)
    WHY: FRIDAY's approach of generating tasks from easy to hard and
    accumulating skills through trial-and-error is elegant. JARVIS should
    autonomously practice and improve on weak areas.

13. DAG-BASED TASK ORCHESTRATION (UFO³ Galaxy)
    WHY: Complex multi-step workflows need parallel execution with
    dependencies. UFO³'s TaskConstellation model with dynamic DAG updates
    enables adaptive, resilient workflow execution.

14. ACCESSIBILITY TREE + VISION HYBRID (UFO² / OmniParser)
    WHY: Using accessibility trees when available and falling back to
    vision (OmniParser) provides the best of both worlds. JARVIS should
    always prefer structured data but degrade gracefully to vision.

15. FTS5 + KNOWLEDGE GRAPH HYBRID MEMORY (Hermes + Hindsight)
    WHY: Combining full-text search (fast keyword lookup) with knowledge
    graphs (entity relationships + synthesis) provides comprehensive
    recall. JARVIS should use both for different query types.
